# Plan 02-01: Core Stability Fixes

## Objective

Fix five confirmed runtime bugs so a full 6-shot round can complete without null reference exceptions, unhandled async exceptions, or iOS Safari memory crashes: cap the leaderboard retry queue (STAB-01), add outer exception safety to async fire-and-forget handlers (STAB-02), guard `Camera.main` in ScoringManager (STAB-03), add diagnostic logging for camera tag lookup failures in CameraController (STAB-04), and set the physics fixed timestep to 60Hz (STAB-05).

## Context

@.planning/phases/02-runtime-stability/02-CONTEXT.md
@.planning/REQUIREMENTS.md

Key decisions from CONTEXT.md:
- D-02: Retry queue capped at 10 entries; entries beyond cap are dropped
- D-03: Null guards for Camera.main and tagged lookups must emit Debug.LogWarning
- D-04: All async void fire-and-forget methods must have outer try-catch wrapping entire method body; exceptions logged with Debug.LogError, not rethrown
- D-06: Set Time.fixedDeltaTime via ProjectSettings/TimeManager.asset (m_FixedTimestep: 0.01667 = 1/60)

## Tasks

### Task 1: Cap leaderboard retry queue at 10 entries (STAB-01)
**type:** auto
**file:** `Assets/Scripts/Multiplayer/LeaderboardManager.cs`

In `PostScoreWithRetryAsync`, before enqueueing to `retryQueue`, check if `retryQueue.Count >= 10`. If so, log a `Debug.LogWarning` and return without enqueuing. Add a `private const int MaxRetryQueueSize = 10;` constant at class level.

**Done criteria:**
- `MaxRetryQueueSize = 10` constant exists
- `PostScoreWithRetryAsync` drops entries when queue is at capacity
- Warning is logged when entry is dropped

### Task 2: Add outer try-catch to async void handlers (STAB-02)
**type:** auto
**file:** `Assets/Scripts/Multiplayer/LeaderboardManager.cs`

The existing `HandleBestDistanceUpdated` and `HandleGameOver` async void methods already have try-catch — verify both are fully wrapped. Also check `Start()` (async void) — it already has try-catch. Add `this == null` guard checks after each `await` point in `Start()`, `HandleBestDistanceUpdated`, and `HandleGameOver` per decision D-05.

**Done criteria:**
- All `async void` methods in LeaderboardManager have entire body wrapped in try-catch
- `this == null` check after each await point in those methods

### Task 3: Guard Camera.main in ScoringManager.HandleBallLanded (STAB-03)
**type:** auto
**file:** `Assets/Scripts/Environment/ScoringManager.cs`

Line 179: `ShotPopup.Create(landPosition + Vector3.up * 1.5f, distanceToPin, Camera.main);`

Add a null check before calling `ShotPopup.Create`. If `Camera.main` is null, log `Debug.LogWarning("[ScoringManager] Camera.main is null — skipping shot popup")` and skip the popup creation.

**Done criteria:**
- `Camera.main` is retrieved into a local variable and null-checked before use
- `Debug.LogWarning` with descriptive message is emitted on null
- `ShotPopup.Create` is not called when camera is null

### Task 4: Add diagnostic logging for camera tag lookup failures (STAB-04)
**type:** auto
**file:** `Assets/Scripts/Camera/CameraController.cs`

In `Start()`, the three `GameObject.FindWithTag` calls for `TeeCamera`, `FlightCamera`, and `LandingCamera` silently return null. After each tag lookup, add a `Debug.LogWarning` if the result is null. Format: `[CameraController] TeeCamera tag lookup returned null — camera transitions will be disabled`.

**Done criteria:**
- Each camera tag lookup is followed by a null check that emits `Debug.LogWarning` with the tag name
- `ActivateTeeCamera()` call at end of `Start()` is guarded by checking if `teeCamera != null` (or gracefully handles null in that method)

### Task 5: Set physics fixed timestep to 60Hz (STAB-05)
**type:** auto
**file:** `ProjectSettings/TimeManager.asset`

Change `m_FixedTimestep: 0.02` to `m_FixedTimestep: 0.01666667` (1/60 = 0.01666...).

**Done criteria:**
- `TimeManager.asset` has `m_FixedTimestep: 0.01666667`
